description: Open a new entry in the local task tracker (tasks/ directory). Use when starting a non-trivial piece of work the user has asked for.
argument-hint: <short title> [| area]
---

Create a new task file in `tasks/` for the work you are about to start.

Argument: `$ARGUMENTS` — a short title, optionally followed by ` | <area>` (e.g. `combat`, `vfx`, `balance`, `ui`, `meta`). If no area is given, infer one from the work and pick a sensible single-word value.

Steps:

1. List existing files in `tasks/` (Bash `ls tasks/`). Find the highest 3-digit `NNN-` prefix and pick the next ID, zero-padded to 3 digits.
2. Slugify the title: lowercase, spaces → `-`, drop punctuation, max ~6 words. The filename is `tasks/NNN-<slug>.md`.
3. Get today's date in `YYYY-MM-DD` format (Bash `date +%Y-%m-%d`).
4. Write the task file with this exact frontmatter and body skeleton:

   ```markdown
   ---
   id: "NNN"
   title: <full title from $ARGUMENTS, before the `|`>
   status: active
   area: <area>
   priority: normal
   started: <today>
   finished:
   ---

   ## Description

   <one or two sentences in your own words on what this task is, based on the user's request and conversation context. Do NOT just restate the title.>

   ## Work log

   - <today>: opened.

   ## Summary

   _(filled in at /task-done)_
   ```

5. Tell the user one short line: `Opened task NNN — <title> (area: <area>).` Do not show the file contents back. The HTML viewer will refresh automatically via the PostToolUse hook.

Notes:
- Granularity is coarse — one task per user request, not per subtask. Use `TodoWrite` for in-session subtasks.
- Do NOT manually edit `tasks/TASKS.md` or `tasks/tasks.html`. Both are generated by `tasks/build_tasks_html.py` and overwritten on every edit.
- If the user has not yet given a clear deliverable, do NOT open a task — ask them to clarify scope first.
